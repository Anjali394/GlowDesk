# System Architecture Document (SAD)

# GlowDesk — Salon Management Platform

**Version:** 1.0 (MVP)

---

## 1. Overview

GlowDesk is a monolithic Spring Boot 3 REST API with clean layered architecture. The MVP is intentionally monolithic — designed with clear separation of concerns so it can evolve into microservices as the business grows.

---

## 2. Architectural Goals

- Clean layered architecture (Controller → Service → Repository)
- RESTful APIs versioned under `/api/v1/`
- Stateless authentication using JWT
- PostgreSQL persistence (Neon managed cloud)
- Cloud deployment on Railway with zero-downtime deploys
- RFC 7807 Problem Details for standard error responses
- High concurrency via Java 21 virtual threads
- Future-ready for multi-branch salons

## 4. Layered Design

### Presentation Layer
- REST Controllers (`@RestController`)
- Request/Response DTOs (immutable records or Lombok `@Data`)
- Input validation via Jakarta Bean Validation (`@Valid`, `@NotNull`, `@Size`)
- RFC 7807 Problem Details error responses (built-in, `spring.mvc.problemdetails.enabled: true`)
- API versioning via URL prefix `/api/v1/`
- Swagger UI auto-generated by springdoc-openapi 2.x at `/swagger-ui/index.html`

### Service Layer
Contains all business logic:
- Appointment booking (time conflict check, duration calculation)
- Stylist availability and auto-assign logic
- Status transition enforcement
- Cancellation eligibility check (1-hour rule)
- Notification triggering at lifecycle events

### Repository Layer
- Spring Data JPA (Hibernate 6)
- Flyway migrations own the schema (`ddl-auto: validate`)
- UUID primary keys via `gen_random_uuid()`
- Soft-delete via `is_active` flag on business entities

### Security Layer
- Spring Security 6 filter chain
- `JwtAuthenticationFilter` validates token on every protected request
- Role-based access control via `@PreAuthorize`
- BCrypt password hashing on registration
- Three roles: CUSTOMER, RECEPTIONIST, ADMIN

### Notification Module
- In-app only for MVP (stored in `notification` table)
- Triggered from service layer at key booking lifecycle events
- Future: pluggable interface for email (Mailgun) / SMS (Twilio)

### Scheduler Module
- Spring `@Scheduled` task, runs every 60 seconds
- Finds `PENDING` appointments where `expires_at < now()`
- Transitions status to `EXPIRED`
- Triggers customer notification

## 6. Java 21 Virtual Threads

Virtual threads are enabled via:

```yaml
spring:
  threads:
    virtual:
      enabled: true
```

This tells Spring Boot to use virtual threads for all incoming HTTP requests. Since most time in a booking API is spent waiting on the database (blocking I/O), virtual threads allow thousands of concurrent requests without thousands of OS threads.

**No code changes required** — Spring Boot handles this automatically when the config flag is set.

## 7. Authentication Flow

```
POST /api/v1/auth/register  →  Create User + Role + Customer profile
POST /api/v1/auth/login     →  Validate credentials → Issue JWT

Protected request:
  Client sends: Authorization: Bearer <token>
  JwtAuthFilter validates token → loads UserDetails → sets SecurityContext
  Controller executes → role checked via @PreAuthorize
```

## 8. Booking Flow

```
Customer → POST /api/v1/appointments
  ├── Resolve services (or expand combo → services)
  ├── Calculate total duration (sum of service durations)
  ├── Calculate end_time = start_time + total_duration
  ├── Validate within branch operating hours
  ├── Auto-assign stylist (highest rated available)
  │     └── Check: no overlapping appointment in time window
  ├── Capture price_at_booking per service
  ├── Create Appointment (PENDING) + AppointmentService rows
  ├── Set expires_at = now + 30 min
  └── Trigger notifications → Customer + Receptionist

Receptionist → PATCH /api/v1/appointments/{id}/confirm
  └── PENDING → CONFIRMED → Notify Customer

Receptionist → PATCH /api/v1/appointments/{id}/reject
  └── PENDING → REJECTED → Notify Customer

Receptionist → PATCH /api/v1/appointments/{id}/complete
  └── CONFIRMED → COMPLETED → Notify Customer

Scheduler (every 60s) → find PENDING where expires_at < now
  └── PENDING → EXPIRED → Notify Customer

Customer → PATCH /api/v1/appointments/{id}/cancel
  ├── Allowed if PENDING (any time)
  ├── Allowed if CONFIRMED AND start ≥ 1 hr away
  └── → CANCELLED → Notify Receptionist
```

## 9. API Versioning

All endpoints prefixed with `/api/v1/`. Breaking changes introduce a new prefix (`/api/v2/`), not header-based versioning.

Example endpoints:
```
POST   /api/v1/auth/register

GET    /api/v1/services
POST   /api/v1/appointments
PATCH  /api/v1/appointments/{id}/confirm
PATCH  /api/v1/appointments/{id}/complete
POST   /api/v1/appointments/{id}/rate
GET    /api/v1/notifications
```

## 10. Error Handling

Spring Boot 3's built-in RFC 7807 Problem Details is enabled via:

```yaml
spring:
  mvc:
    problemdetails:
      enabled: true
```

Standard error response:
```json
{
  "type": "about:blank",
  "title": "Bad Request",
  "status": 400,
  "detail": "No available stylist for the requested time slot",
  "instance": "/api/v1/appointments"
}
```

Custom exceptions (`ResourceNotFoundException`, `BusinessRuleException`) are mapped to the correct HTTP status via `@ControllerAdvice` where fine-grained control is needed.

## 11. Configuration

`application.yml`:

```yaml
spring:
  datasource:
    url: ${DB_URL}
    username: ${DB_USERNAME}
    password: ${DB_PASSWORD}
  jpa:
    hibernate:
      ddl-auto: validate       # Flyway owns the schema
    show-sql: false
    properties:
      hibernate:
        format_sql: true
  flyway:
    enabled: true
  mvc:
    problemdetails:
      enabled: true             # RFC 7807 error responses
  threads:
    virtual:
      enabled: true             # Java 21 virtual threads

jwt:
  secret: ${JWT_SECRET}
  expiration: ${JWT_EXPIRATION:86400000}

server:
  port: 8080

springdoc:
  swagger-ui:
    path: /swagger-ui.html
  api-docs:
    path: /api-docs

management:
  endpoints:
    web:
      exposure:
        include: health, info   # Actuator for Railway health check
```

## 12. Testing Strategy

| Layer | Type | Tool | Notes |
|---|---|---|---|
| Service | Unit | JUnit 5 + Mockito | Mock repositories, test business rules |
| Repository | Integration | Testcontainers + `@ServiceConnection` | Real PostgreSQL in Docker |
| Controller | Slice | MockMvc + `@WebMvcTest` | Test HTTP layer in isolation |

`@ServiceConnection` (Spring Boot 3.1+) auto-wires the Testcontainers PostgreSQL container — no manual URL config needed:

```java
@Container
@ServiceConnection
static PostgreSQLContainer<?> postgres = new PostgreSQLContainer<>("postgres:16");
```

## 13. Deployment Pipeline

```
Developer → git push → GitHub (main branch)
  → Railway detects push → pulls repo
  → Docker build (Dockerfile)
  → Runs Flyway migrations on Neon PostgreSQL
  → Spring Boot starts
  → Railway health check: GET /actuator/health
  → Traffic routed to new deployment
```

## 15. Future Evolution

As load grows, the monolith can be split into microservices:

- Auth Service
- Appointment Service
- Stylist Service
- Notification Service (Kafka consumer)
- Analytics Service

Infrastructure additions:
- **Redis** — cache service catalog and stylist availability
- **Kafka** — event streaming for notifications and analytics
- **API Gateway** — unified entry point for microservices
